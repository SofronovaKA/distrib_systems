## Вариант 17

### Тема: gRPC-сервис Calendar (Unary RPC)

---

## 📌 Задание

Реализовать gRPC-сервис **Calendar**, предназначенный для управления событиями. Основной функционал включает:

* Метод **CreateEvent(EventDetails)** — создание нового события в календаре.
* Тип взаимодействия: **Unary RPC** (одиночный запрос от клиента и одиночный ответ от сервера).

В сообщении `EventDetails` передаются параметры события: заголовок, описание, дата и время начала/окончания.

---

## 🏗 Архитектура решения

В работе реализована стандартная архитектура взаимодействия между клиентом и сервером по протоколу gRPC.

+-------------+         gRPC          +-------------+
|   Клиент    |  <--------------->    |   Сервер    |
|  client.py  |       (Unary)         |  server.py  |
+-------------+                       +-------------+
|
|
+---------------+
|  events_list  |
|  (в памяти)   |
+---------------+

---

### Компоненты системы

* **Client (`client.py`)**: Формирует объект события и отправляет его серверу, ожидает подтверждение создания.
* **Server (`server.py`)**: Принимает данные, валидирует их и сохраняет в локальный список (имитация БД).
* **`calendar.proto`**: IDL-файл (Interface Definition Language), описывающий структуру сервиса и сообщений.
* **`events_list`**: Хранилище в оперативной памяти сервера для накопления созданных событий.

---

## 🧩 Описание реализации

Основная логика сервиса описана в файле:
`calendar.proto`

Ключевой метод `CreateEvent` принимает структуру `EventRequest` и возвращает `EventResponse` со статусом операции и ID созданного события. Поскольку это **Unary RPC**, соединение закрывается сразу после получения ответа.

## ⚙ Генерация gRPC-кода

Для работы с gRPC на Python была выполнена генерация интерфейсных файлов с помощью компилятора `protoc`. Команда:
`python -m grpc_tools.protoc -I. --python_out=. --grpc_python_out=. calendar.proto`

После выполнения команды в директории появились файлы:
`calendar_pb2.py` (содержит классы сообщений);
`calendar_pb2_grpc.py` (содержит классы клиента и сервера).

---

## 🖥 Реализация сервера

Сервер реализован в файле `server.py`
Был создан класс CalendarService, который переопределяет метод CreateEvent.

Логика метода:

- Получение данных из запроса.
- Генерация уникального идентификатора для события.
- Добавление события в глобальный список events.
- Логирование действия в консоль сервера.
- Возврат успешного статуса клиенту.

Сервер слушает входящие соединения на порту 50051.

## 💻 Реализация клиента

Клиентская часть `client.py` выполняет следующие шаги:

- Создает канал связи с localhost:50051.
- Инициализирует заглушку (stub) сервиса.
- Собирает данные для нового события (название, время, описание).
- Вызывает удаленный метод CreateEvent и выводит результат на экран.

## 📊 Результат работы

В ходе тестирования я проверила работу сервиса, создав несколько различных событий. Сервер успешно принимает запросы, корректно обрабатывает структуру EventDetails и возвращает уникальные ID событий. Взаимодействие происходит мгновенно, данные передаются в бинарном формате, что эффективнее традиционного REST. Все созданные события сохраняются в оперативной памяти сервера до его перезапуска.
